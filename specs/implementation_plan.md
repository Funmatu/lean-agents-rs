# 【絶対厳守】ASUS GX10(UMA) 最適化 Qwen3.5-27B マルチエージェント・システム開発仕様書（Web検索統合版）

## 1. プロジェクトの最終目的とあなたの役割
あなたは世界最高峰のRustアーキテクト兼AIシステムエンジニアです。
本プロジェクトの目的は、単一のハードウェア上で4つの役割を持つAIエージェント群が自律的に協調し、高品質なコード生成・設計・レビューを行う「非同期コンテキストグラフ・システム」をRustでフルスクラッチ開発することです。
既存のLangGraphのような肥大化したフレームワークは一切使用せず、Rustの型システムを活用した独自の状態遷移機械（State Machine）を構築します。実用的なハルシネーション対策として、非同期のWeb検索（Tool Call）機能をネイティブに統合します。

## 2. 実行環境とハードウェア制約（暗黙知の排除）
本システムは、以下の特殊なハードウェア環境での稼働を大前提とします。この制約を無視した設計（無駄なメモリ確保、無駄なデータコピー、コンテキストの肥大化）は絶対に許容されません。

* **ハードウェア:** ASUS Ascent GX10 (NVIDIA DGX Spark同等 / GB10)
* **アーキテクチャ:** ARM64 CPU + 単一Blackwell GPU
* **メモリ構造:** **128GB LPDDR5x UMA (Unified Memory Architecture)**
    * *絶対遵守事項:* CPUとGPUは128GBの物理メモリを共有し、帯域幅は273 GB/sです。Rust側で巨大な文字列データを不必要に複製（`clone`）したり、オンメモリに溜め込んだりすると、SGLang（GPU側）の推論帯域を奪い、システム全体のTPSが致命的に低下します。
* **バックエンド推論エンジン:** SGLang (OpenAI互換API)
* **ターゲットLLM:** Qwen3.5-27B
    * *絶対遵守事項:* SGLangの「RadixAttention（Prefix Caching）」機能を最大限に活用することが本アーキテクチャの要です。共通のプロンプト部分は必ずメッセージ配列の先頭（不変のPrefix）に配置してください。

## 3. Web検索と「揮発性コンテキスト (Volatile Context)」の概念
エージェントがWeb検索（Tool Call）を行った際、取得した検索結果（HTMLスニペット等）を `Message` 履歴に永続化してはなりません。履歴に永続化すると、ターンが進むごとにQwen3.5-27BのKVキャッシュが爆発し、UMAのメモリ帯域を食いつぶします。
検索結果は「直後の1ターンの推論でのみ参照される『揮発性コンテキスト』」として扱い、状態遷移が進んだ瞬間にRust側でメモリから破棄（Drop）するガベージコレクション機構を必ず設計してください。

## 4. エージェントの役割と厳密な制約
以下の4つのエージェントを定義し、それぞれにシステムプロンプトとして以下の制約を**ハードコード**してください。必要に応じてエージェントは指定のJSONフォーマットで `{"action": "search", "query": "検索語"}` を出力し、外部情報を取得できます。

1.  **Orchestrator (Leader)**
    * **役割:** 計画、タスク分割、進捗管理、最終決定。
    * **制約:** 自身ではコードを書かない。Delegate Modeに徹する。
2.  **Architect (Designer)**
    * **役割:** 技術選定、データ構造、インターフェース設計。
    * **制約:** 具体的な機能実装には関与しない。未知のライブラリ仕様は必ずWeb検索(`action: search`)を実行して裏付けを取ること。
3.  **Programmer (Executor)**
    * **役割:** 設計書に忠実に従い実務を実行。
    * **制約:** 設計の意図を勝手に変更しない。実装上のAPI仕様が不明な場合はWeb検索を実行すること。
4.  **DevilsAdvocate (QA / Critic)**
    * **役割:** 破壊的な批判、セキュリティリスクの指摘、エッジケースの提示。
    * **制約:** 建設的な代替案をセットで提示すること。単なる否定で終わらせないこと。

## 5. 開発プロセス原則 (SDD & TDD)
コード生成時、以下のステップを厳格に守り、各ステップの完了ごとにユーザーに報告して次の許可を得ること。一気にすべてを実装してはなりません。
* **ルール1:** ロジックの前に `struct`, `enum`, `trait` で状態と型を定義。
* **ルール2:** テスト（単体テスト、モックを用いた通信テスト）を先に書き、それが通るように実装。
* **ルール3:** `unwrap()`, `expect()` はテストコード以外で使用禁止。必ずカスタムエラー型でハンドリングすること。

## 6. 実装ステップ詳細

### 【Step 1】 ドメインモデリングと状態定義（Tool Call統合）
* `serde` を用い、以下のデータ構造を定義せよ。
    * `enum AgentRole` (4つの役割)
    * `struct Message` (送信者、内容)
    * `struct ToolCallRequest` (アクション名、クエリパラメータ等)
    * `enum WorkflowState` (Init, Planning, Designing, Implementing, Reviewing, **ToolCalling**, Completed, Escalated)
    * `struct ContextGraph`
        * 全メッセージ履歴(`Vec<Message>`)
        * 現在の状態(`WorkflowState`)
        * イテレーションカウンタ
        * **`volatile_context: Option<String>` (揮発性の検索結果保持領域。イテレーションやフェーズが切り替わる際に必ず `None` にリセットされること)**
* **要件:** 各状態間の正しい遷移（ToolCallingからは呼び出し元の状態へ戻る等）と、`volatile_context` が正しくクリアされることを検証する単体テストを書け。

### 【Step 2】 UMA/SGLang最適化 LLMクライアント ＆ Web Searchモジュール
* `reqwest` を用い、SGLang互換エンドポイントを叩く `async_trait LlmClient` を実装せよ（ストリーミング受信推奨）。
* `reqwest` を用い、Tavily API (または SearXNG 等) を非同期で叩く `async_trait SearchClient` を実装せよ。
* **要件:** テスト用に両方のMockを実装し、通信エラー時のリトライロジックを検証するテストを書け。

### 【Step 3】 構造化出力（Structured Output）とTool Callパーサーの実装
* エージェントの出力から「通常の発言(Approve/Reject含む)」か「Tool Call要求(`{"action": "search", "query": "..."}`）」かを判別する厳格なパーサーモジュールを実装せよ。
* **要件:** JSON Schemaを強制するか、正規表現等を用いて安全にデシリアライズするテストを書け。

### 【Step 4】 エージェント・トレイトの実装とプロンプトバインディング
* `Agent` トレイトを定義し、4つのエージェントを実装せよ。
* **要件:** 第4項の「絶対制約」をシステムプロンプトとして埋め込むこと。
* **要件:** SGLangへリクエストを送る際、`ContextGraph` の `volatile_context` に値が存在する場合は、それを「一時的な参考情報」としてシステムプロンプトの直後（ただしPrefixキャッシュを壊さない位置）に挿入してプロンプトを構築するロジックを実装せよ。



### 【Step 5】 コンテキストグラフ実行エンジン（デッドロック防止機構）
* `ContextGraph` の状態を評価し、次にどのアクターまたは外部ツールを呼び出すか決定する非同期ループを実装せよ。
* **要件（ツールコール・インターセプト）:** エージェントの出力がTool Call要求だった場合、状態を `WorkflowState::ToolCalling` に遷移させ、非同期でWeb検索を実行。結果を `volatile_context` に格納し、再度同じエージェントを呼び出せ。エージェントが最終的な回答を出力した段階で `volatile_context` を破棄せよ。
* **要件（デッドロック防止）:** 承認(Approve)が出ないまま規定回数（例: `MAX_ITERATIONS = 3`）に達した場合、強制的に `WorkflowState::Escalated` へ移行させループを抜けよ。
* **要件:** Mockを用いて、「検索を経てからApproveを出すパス」と「Escalatedで安全に停止するパス」のE2Eテストを書け。

### 【Step 6】 メインループとCLI統合
* `tokio` ランタイムを用い、環境変数からSGLangのAPI URL、検索APIのキー等を読み込む `main.rs` を実装せよ。
